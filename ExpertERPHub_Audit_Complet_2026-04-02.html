3. Détails des problèmes critiques

#1 — Forge de session (portal-auth.js)

Le système de session utilise localStorage sans aucun signing cryptographique. N'importe qui peut exécuter dans la console :

localStorage.setItem('experterphub_portal_session', JSON.stringify({role:'admin', email:'fake@admin.com', expiresAt:'2026-12-31'}))

Impact : accès complet au dashboard admin, modification de statuts consultants/entreprises.

✓ Fix : utiliser les sessions natives Supabase Auth avec JWT signé côté serveur.

#2 — Spoofing email messagerie (messaging.js)

La fonction send() accepte senderEmail du client sans vérification serveur. Un utilisateur malveillant peut envoyer des messages en se faisant passer pour n'importe qui.

Impact : usurpation d'identité, phishing interne, fraude.

✓ Fix : ajouter une politique RLS vérifiant que sender_email = auth.email() sur la table messages.

#3 — isAdminEmail() cassée (utils.js)

La version synchrone utilise crypto.subtle.digest (async) mais retourne le résultat avant la résolution de la Promise. Elle retourne toujours false.

Impact : tout code utilisant la version synchrone ne détecte jamais un admin.

✓ Fix : supprimer la version synchrone, utiliser exclusivement isAdminEmailAsync().

#4 — Filtre B2B absent (consultant.html)

La page profil charge et affiche les ressources B2B (import_b2b) sans vérifier si l'utilisateur est une entreprise. Un visiteur peut accéder à un profil B2B privé via URL directe.

Impact : violation de la règle business — les ressources B2B doivent être totalement cachées aux non-entreprises.

✓ Fix : ajouter un check isB2BResource() + isEnterprise avant renderConsultant(). Rediriger sinon.

#5 — CONSULTANT_COLS non défini (index.html)

La variable CONSULTANT_COLS est utilisée dans 4 fonctions async mais n'est jamais déclarée dans index.html. Ces fonctions crashent avec ReferenceError.

Impact : loadConsultants(), submitPortalAccess(), validateInviteCode(), syncLegacyHashesByEmail() sont cassées.

✓ Fix : définir CONSULTANT_COLS en haut du script, ou utiliser window.CONSULTANT_COLS avec fallback.

#6 — Fonctions manquantes (dashboard-entreprise.html)

buildResourceCode() et generateInternalResourceEmail() sont appelées lors de l'import CSV/Excel et de l'ajout manuel de ressources, mais ne sont définies nulle part dans ce fichier.

Impact : l'import de ressources et l'ajout manuel sont complètement cassés.

✓ Fix : copier ces fonctions depuis index.html ou créer un fichier utils partagé.

4. Détails des problèmes de sévérité haute

#7 — XSS via onclick (dashboard-entreprise.html)

L'attribut onclick utilise une interpolation de string avec des données utilisateur (email, nom). Un email malicieux comme '); alert('xss'); // pourrait exécuter du JavaScript.

✓ Fix : utiliser data- attributes et addEventListener au lieu de onclick inline.

#8 — Email dans l'URL (dashboard-consultant.html)

L'email est passé en paramètre URL (?email=...) pour le reset password. Cela expose l'email dans l'historique du navigateur, les logs serveur, et les headers Referer.

✓ Fix : utiliser sessionStorage ou POST au lieu d'un paramètre URL.

#9 — Hash admin email public (config.js)

Le hash SHA-256 de l'email admin est visible dans le code source public. Un attaquant peut faire un reverse-lookup via rainbow tables pour découvrir l'email admin.

✓ Fix : vérifier le statut admin côté serveur via Supabase RLS, pas côté client.

#10 — Pas de CSRF (dashboard-admin.html)

Les actions admin (changement de statut, approbation) n'ont aucun token CSRF. Une page malveillante pourrait déclencher ces actions.

✓ Fix : ajouter une validation CSRF sur toutes les actions qui modifient des données.

#11 — Clé API dans les logs (api/send-email.js)

En cas d'erreur, console.error(data) pourrait exposer la clé RESEND_API_KEY dans les logs Vercel, accessibles à toute l'équipe.

✓ Fix : masquer les données sensibles dans les logs d'erreur.
